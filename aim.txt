# Project Specification - Business Empire Prototype V0.1

## Objective

Build a browser-based single-player business simulation prototype to validate the core gameplay loop.

This is **NOT** the final game. The objective is to determine whether building, optimizing, and expanding companies is enjoyable before adding multiplayer and alliance systems.

---

* Next.js (App Router)
* TypeScript
* Tailwind CSS
* React

Backend

* Supabase
* PostgreSQL

Hosting

* Vercel

Architecture Requirements

* Clean Architecture
* Modular Design
* Reusable Systems
* Future Multiplayer Ready

---

# Primary Goal

The player should experience this gameplay loop:

Build → Produce → Sell → Earn Money → Upgrade → Expand → Merge Companies → Repeat

The codebase should be scalable enough to later support multiplayer without major rewrites.

## Clarifications for Prototype V0.1

### 1. Simulation Tick Rate

Use a **1-second game tick**.

Every second:

* Update production
* Consume required resources
* Generate finished products
* Calculate income
* Update expenses
* Refresh UI

The simulation should **only run while the game is open**.

If the browser tab is inactive or closed:

* Pause the simulation.
* Do not calculate offline progress.

Offline progression can be added in a future version.

---

### 2. Building Placement Rules

For the prototype:

* Buildings cannot overlap.
* Buildings must stay completely inside the map.
* Rotation is **not required**.
* Buildings may be moved freely.
* Buildings may be demolished at any time.
* Demolishing returns **50%** of the building cost.
* No adjacency bonuses.
* No spacing rules.
* Ignore roads and logistics connections.

The goal is to validate gameplay, not city-building mechanics.

---

### 3. Economy & Balancing

You may design reasonable placeholder formulas.

The formulas should be modular and easy to modify later.

Examples:

* Production Rate
* Upgrade Cost
* Maintenance Cost
* Resource Consumption
* Company Value

Do **not** hardcode values throughout the project.

Keep all balancing values in centralized configuration files.

---

### 4. Company Merge

Prototype merge:

Requirements:

* 5 Retail Shops
* 1 Factory
* 1 Marketing Office

When merged:

* Remove required buildings.
* Create one Clothing Brand.
* Preserve company statistics.

No animations required.

---

### 5. Company Valuation

Use a simple placeholder formula.

Example:

Company Value =
(Revenue × 30)
+
(Profit × 60)
+
(Level × 1000)
+
(Skill Points × 500)

The formula only needs to demonstrate the system.

---

### 6. Save System

Use Supabase.

For Version 0.1:

* Anonymous single-player save is sufficient.
* One save slot per player.
* Auto-save every 30 seconds.
* Manual save button is optional.

No authentication system is required yet.

---

### 7. UI

Keep the interface simple.

Prioritize:

* usability
* readability
* debugging

Placeholder icons and simple shapes are completely acceptable.

---

### 8. Overall Goal

The objective is **not** to build a polished game.

The objective is to validate whether the gameplay loop:

Build → Produce → Sell → Upgrade → Merge → Expand

is enjoyable enough to justify further development.


Don't optimize for graphics or polish. Optimize for clean architecture and gameplay iteration. Every gameplay value should be easy to rebalance without changing the code.

## Building Footprints (Prototype V0.1)

Map Size:

* 10 × 10 Grid (100 Tiles)

Building Sizes:

| Building                        | Size  |
| ------------------------------- | ----- |
| Iron Mine                       | 2 × 2 |
| Coal Mine                       | 2 × 2 |
| Cotton Farm                     | 2 × 2 |
| Factory                         | 3 × 3 |
| Warehouse                       | 2 × 2 |
| Marketing Office                | 2 × 2 |
| Retail Shop                     | 1 × 1 |
| Clothing Brand (Merged Company) | 4 × 4 |

Notes:

* Buildings cannot overlap.
* Buildings can be moved.
* Buildings can be demolished.
* No rotation is required.
* No adjacency bonuses.
* No roads.
* No spacing restrictions.
* The merge should replace the required buildings with a single Clothing Brand building.


NOT to hardcode these values.

Ask it to create a configuration file like:

/buildingConfig.ts

{
  factory: {
    width: 3,
    height: 3,
    cost: ...
  }
}

That way, if next week you decide the Factory should be 4×3 instead of 3×3, you only change one file instead of searching through the entire codebase.
